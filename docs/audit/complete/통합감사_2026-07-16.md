# 통합 감사 — 연애 DNA v2.1 프로젝트 (Phase 0~4) 코드/문서 정합성 점검

**조사 방식:** 전부 읽기 전용(grep/Read/git log/git diff, 2개 Explore 서브에이전트 병렬 활용). 어떤 파일도 수정하지 않았다.
**범위:** Phase 0~4에서 만든 v2.1 엔진 전체 + 기존 코드와의 경계.

## 요약

- 발견한 문제: **4건** (아래 §1~§5 매핑 기준)
- 이 중 **"즉시 사용자 체감 버그로 이어질 수 있는 것": 2건** — (1) 커플 DNA% 결과를 실제로 보여주는 화면이 어디에도 없고 두 파트너 간 결과가 비대칭으로만 저장됨, (2) `sBase`(ON 경로에서는 `dna_pct`)를 소비하는 티어/코칭 화면들이 옛 분포(mean 70, sd 7.81) 기준으로 보정된 임계값을 그대로 쓰고 있어 새 분포(mean~75, sd~9.7)의 점수를 받으면 등급·문구가 의도와 다르게 느껴질 수 있음
- 나머지 2건(고아 함수, 미기재 테스트 파일 10개)은 사용자에게 보이지 않는 낮은 심각도
- 비목표 영역(Aura, MoodFeed, Couple Wrapped, VIP, twinResponseEngine.ts) 회귀는 **전혀 발견되지 않음** — 전부 깨끗함
- Phase 0 금지 12종 + "폐기 예정 7종" 조기 수정 흔적도 **전혀 없음**

## §1. personaAdapter / computeDnaNationalPercentile / sBase 소비처 전수 확인

### (a) `enneagramCoreToTopType()` 실제 호출 여부
**호출됨 — 고아 아님.** `src/components/GenesisV21Screen.tsx:73`에서 `enneagramCoreToTopType(profile.enneagramCore)`로 실제 호출되며, 결과(`type`, `confidence`)가 바로 다음 줄에서 `computePersonaBlend(type, confidence)`로 이어진다.

### (b) `computeDnaNationalPercentile()` 실제 호출 여부
**호출하는 화면 없음 — 고아 함수.** `src/engine/scoreCalculator.ts`에 정의만 되어 있고 저장소 어디에서도 import/호출되지 않는다. 추가로 확인한 사실: 기존 `computeNationalPercentile()`(구 경로용) 역시 **저장소 어디서도 호출되지 않는다** — "전국 상위 X%" 표시 기능 자체가 현재 앱에 아직 연결된 화면이 없다(둘 다 대기 상태). 버그는 아니지만, Phase 3 보고서가 이미 "대기 상태"라고 남긴 판단이 사실로 재확인됨.

### (c) `scoreStore.sBase` 소비처 전수 (grep 결과)

| 소비처 | 사용 방식 |
|---|---|
| `src/hooks/useMidnightSettlement.ts:35` | `settleMidnight(sTodayOpen, aEndOfDay, store.sBase, ...)` — 자정 정산 시 `sBase`를 "중력 앵커"로 사용(`deltaDecay = -λ·(sTodayOpen - sMasterBase)`) |
| `src/services/weeklyReportService.ts:48` | `[store.sCurrent \|\| store.sBase]` — 주간 리포트 이력 배열의 폴백 시드값 |

| `src/components/AICoachingCard.tsx:35,64,65` | 코칭 카드도 동일한 `displayScore` 패턴으로 `getTierFromScore()` 호출 + 무드태그(`>=70`/`>=40` 하드코딩 임계값) 분기 |

**핵심 발견:** `getTierFromScore()`의 10단계 구간(예: `>=95` "신화적", `>=75` "핑크빛 로맨스" 등)과 `computeNationalPercentile()`의 정규분포 상수(`mean=70, sd=7.81`)는 **구 `S_Base` 분포 전용으로 보정된 값**이다. v2.1 `dna_pct`는 `clamp(75 + 9.7056·z, 50, 100)`(§12 식 10)로 산출되어 **중심(75)과 산포(≈9.7)가 다른 분포**를 가진다. 코드 경로 자체는 멀쩡히 동작하지만(크래시 없음), 같은 숫자값이라도 두 분포에서 "상위 몇 %에 해당하는 점수인지"의 실제 의미가 달라진다 — 예를 들어 dna_pct의 평균값(75)은 구 분포 기준으론 상위권(`getTierFromScore(75)`="🍃 핑크빛 로맨스")에 해당하지만, dna_pct 자체 분포에서 75는 딱 "평균"이다. 즉 ON 경로 유저는 "평균적인 궁합"인데도 화면엔 "평균보다 꽤 좋은 티어"로 표시될 수 있다.

## §2. 파트너 비동시 완료 시나리오

- `computeRomanticDnaV21()` 호출 시점: **자기 자신의 인터뷰가 끝나는 시점에만** 호출된다(`GenesisV21Screen.tsx` 완료 콜백, 상대방의 완료 여부와 무관하게 각자 독립 실행).
- 파트너가 아직 안 끝났을 때: `getPartnerPsychProfile()`이 `.maybeSingle()`로 null을 정상 반환하고, 호출부도 `try/catch` + `if (partnerProfile)` 가드로 감싸져 있어 **크래시/무한로딩 없음** — `dnaResult`는 null 유지, `sBase`는 자기 자신과의 상성 폴백(`generateBaseScore`)으로 정상 대체됨. 이 부분은 깨끗하다.
- **문제는 그다음이다:** `dnaResult`(`coupleStore`)를 읽거나 쓰는 코드는 저장소 전체에서 **`GenesisV21Screen.tsx` 단 한 곳뿐**이다(grep 재확인). 즉:
  1. 현재 앱에 "커플 DNA%를 보여주는 화면" 자체가 **존재하지 않는다.** (지난 Phase 4에서 사용자에게 요청드린 "커플 화면에서 dnaResult 노출 확인" E2E 체크는, 실기기 테스트를 하더라도 애초에 그걸 볼 화면이 없어 실패할 수밖에 없는 상태였다.)
  2. 두 사람이 각자 다른 시점에 인터뷰를 마쳤을 때, **먼저 끝난 사람**의 기기에서는 `dnaResult`가 영원히 null로 남는다(그 사람의 `GenesisV21Screen`은 자기 완료 시점에 파트너가 아직 없어 계산을 못 하고, 온보딩 화면이라 재방문도 안 됨). **나중에 끝난 사람**의 기기에서만 `dnaResult`가 한 번 계산되어 로컬(zustand persist)에 저장되고, 그 값은 서버(`couple_dna_results`, 아직 미사용)나 파트너 기기로 전혀 전파되지 않는다.
  3. 결과: 두 파트너가 서로 다른(한쪽은 null, 한쪽은 값 있음) DNA 상태를 영구히 갖게 되며, 재계산·동기화 트리거가 전혀 없다.

## §3. 서버 영속성 갭의 실사용 영향

- `coupleStore`의 zustand persist 설정(`src/store/coupleStore.ts`)에는 `partialize`가 없다 — 즉 **`dnaResult`는 AsyncStorage에 그대로 영속화되어 앱 재시작 후에도 남는다**(사라지지 않음). 다만 §2에서 확인했듯 애초에 그걸 보여주는 화면이 없어 실효성이 없다.
- 재계산 트리거: 없음. 로컬 데이터가 초기화(재설치, 캐시 삭제 등)되면 다시 만들 방법은 "온보딩 인터뷰 전체를 처음부터 재수행"뿐이다 — 별도의 "내 DNA% 다시 계산하기" 같은 액션은 존재하지 않는다.
- `couple_dna_results` 테이블 미사용은 **의도적 보류로 확인됨** — (1) 마이그레이션 파일 자체 주석("Phase 1 — 이 테이블은 아직 어떤 코드에서도 읽거나 쓰지 않는다"), (2) Phase 3 완료보고서가 이미 명시적으로 갭으로 기록, (3) 이번 대화에서 사용자가 두 차례 "계속 보류"라고 재확인 — **단순 누락이 아니라 문서화된 의도적 지연**이다.

## §4. 비목표 영역 회귀 여부 — 전부 깨끗함

Explore 서브에이전트로 3갈래 확인, 전부 문제 없음:

2. **MoodFeed/Couple Wrapped/VIP**: "MoodFeed"라는 이름의 파일 자체가 존재하지 않음(무관한 `partnerMoodService.ts`만 존재). `wrappedService.ts`/`WrappedModal.tsx`/`vipPromotionService.ts`/`iapService.ts`/`useVipReconcile.ts` 전부 최근 커밋이 7/10~7/11(v2.1 프로젝트 시작일 7/15보다 최소 5일 전)이라 이번 작업과 무관 확인.
3. **twinResponseEngine.ts**: diff 0줄, 최근 커밋도 7/5(무관). **문서 전제 정정:** 이 파일은 실제로 `PersonaBlend`/`.blend`를 전혀 참조하지 않는다 — 실제 소비처는 `app/(tabs)/chat.tsx`(`buildPersonaBlendPromptSection(personaMatrix.blend)`)이며, 이 파일도 diff 0줄·최근 커밋 7/13으로 무관 확인. ON/OFF 두 경로 모두 동일한 미수정 `computePersonaBlend()`를 호출하고 동일한 `UserPersonaMatrix` 형태를 채우므로 타입/형태 불일치 위험도 없음.
